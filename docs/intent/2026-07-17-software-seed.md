# Intent distillation: software seed and bridge

- Public record ID: `intent-public-2026-07-17-software-seed`
- Recorded: 2026-07-17
- Status: distilled into proposals; not approved runtime behavior
- Canonical private source: not committed to this public repository

## Owner direction

TOHSENO should become an exemplary way to grow a full-stack continuity app from
one deliberate seed. The application's ongoing history should preserve why it
changed, not only what its current source happens to be. Each owner request
should lead to an explicit new intent and manifest state, implementation
evidence, and a release or rollback result.

A native continuity app may become a contextual authority for a browser session:
the browser displays a QR challenge, the phone confirms the exact connection,
and only a small set of hardcoded request types can be authorized. The browser
should show a simple, standard history of those operations.

Optional external rails may eventually let a declared continuity app perform
narrow actions involving Base, Solana, Hyperliquid, or Robinhood. They must not
be treated as equivalent networks, inherit one universal key, or become required
for local continuity.

TOHSENO should also become teachable: long-form programming and vibe-coding
tutorials can show how intention becomes a manifest, contracts, code, tests, and
a live result. A future local initializer can ask which coding agent will create
the app and install the same rails for that agent.

## Product statement under test

> TOHSENO is a software seed: a way to vibe-code a full-stack continuity app
> with intention, preserve how it evolves, and remain able to own and eject it.

This is a product thesis, not a claim that the compiler, bridge, initializer, or
external rails already exist.

## Boundaries preserved in the distillation

- One application still has one primary repeated action.
- First local value still comes before login or pairing.
- Exact owner intent and end-user continuity content remain private by default.
- Practice identity, owner/update authority, browser delegation, release signing,
  and external-action custody remain separate roles.
- A phone may delegate narrow session authority; it never exports a seed or
  long-lived private key to a QR code, browser, extension, server, or log.
- Request audit records are distinct from continuity events and contain no
  request body or private artifact.
- Smart contracts and providers remain optional, manifest-declared modules.
- No paid resource, DNS action, production credential change, store submission,
  package publication, trade, transfer, or contract deployment is authorized by
  this record.

## Organized work produced from this intent

1. Preserve application evolution as intent → manifest → release lineage.
2. Define a finite operation registry and typed audit records.
3. Decide the phone/browser identity, delegation, recovery, and revocation
   protocol.
4. Exercise one bridge in a reference application that retains offline value.
5. Define a safe, pinned, local agent initializer and an explicit free/community
   product path.
6. Select one exact external operation before scaffolding provider or chain code.
7. Publish tutorials only from tested, owner-approved, sanitized evidence.
