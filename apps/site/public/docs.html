Make the idea exist

curl -fsSL https://tohseno.com/install.sh | bash
tohseno

Choose Take your first shot, Take another shot, or Continue a shot. Every app lands as an independent repository under your configured shots directory—~/tohseno/shots by default. TOHSENO shows only implemented platforms—currently iOS—and launches an installed Codex or Claude Code inside it. The agent reads local instructions, then asks what you want to make.

With the interactive launcher, your private product intention stays in that conversation. For an explicit local input, use one UTF-8 Markdown file and repeat --reference for image references:

tohseno create --file intention.md --reference sketch.png

The file paths are ordinary local inputs; their private contents never enter shell arguments, the factory release, Git history, or centralized logs.

One factory, multiple doors

tohseno studio
tohseno studio --port 4747 --no-open

Studio is a local browser contact sheet for the same configured shots directory used by the CLI. It binds to 127.0.0.1, never the LAN, and watches that directory for shots created through either door. Its creation form accepts a long intention, one optional Markdown file, and optional image references.

The terminal and Studio call the same factory. They share concurrency-safe allocation, the pinned factory release, atomic publication, independent Git history, private provenance, and verification. Studio is not a daemon requirement: close it and the same shots remain available through tohseno list, verify, and open.

What the agent operates

Every new shot pins its operational playbook and machine commands locally. The agent starts a localhost Bun API, initializes its SQLite database, waits for health, configures the Debug iOS endpoint, launches a real simulator when Xcode is available, reads structured logs, verifies changes, and stops only processes owned by that shot.

For a physical Debug device, the agent may explicitly request a Cloudflare Quick Tunnel. Its random public URL is temporary development reachability—not authentication, uptime, or production infrastructure—and is written only to a gitignored Debug configuration file.

Run the real Simulator

tohseno run <shot>
tohseno preview <shot>
tohseno doctor

On macOS with Xcode command-line tools and an available iPhone Simulator, run builds, installs, and launches the shot in Apple Simulator, then attempts a private contact-sheet capture. A failed capture does not stop an otherwise running app.

preview adds an interactive local browser stream of that same native Simulator; it is not an in-browser iOS emulator. Interactive preview additionally requires Apple Silicon, a native arm64 Node.js 20 or newer, and the pinned serve-sim helper. Simulator use does not require paid Apple Developer Program membership. doctor reports the selected Node architecture and exact readiness, and unsupported preview does not disable Studio’s contact sheet, creation, or ordinary CLI verification.

The point

The prototype is the payoff: a real thing on your phone that you can touch, change, keep, or leave on the contact sheet. Most shots miss. That is why you can take a lot of them. TOHSENO makes no promise of riches and never turns financial plumbing into the reason to build.

What stays local

A shot has its own Git history, SwiftUI app, Bun API, SQLite migrations, manifest, tests, landing page, verifier, operations, and release provenance. Writing and identity stay on the device. The baseline API stores operational metadata only and never receives writing content. Development data, logs, PIDs, and generated endpoints are gitignored.

When create --file or Studio receives private creation input, its normalized intention, reference copies, hashes, original filenames, options, and structured events stay in the shot’s gitignored .tohseno/provenance/ directory. Temporary Studio uploads are removed after the job finishes or stops safely. Studio does not upload shots or creation input to TOHSENO; the selected coding agent uses that input under its own provider and privacy settings.

The app’s core action remains useful offline. Seed-phrase identity replaces accounts, and every shot builds without TOHSENO credentials or a live factory.

Approval boundary

The agent can diagnose and prepare locally. It must ask before creating accounts, spending money, handling credentials, changing DNS, publishing packages, deploying production, or submitting to an app store. TOHSENO never runs the prepared fastlane beta lane unprompted.

When you say “put this online,” “ship this,” or “send this to TestFlight,” the agent first performs read-only production inspection, explains the exact blockers, and requests only the authority actually needed.

Current production boundary

Production inspection, stable-HTTPS validation, single-instance SQLite semantics, backup declarations, and unresolved-secret reporting are implemented. Broad VPS provisioning, deployment, monitoring, recovery, DNS changes, and store submission are not implemented. A Quick Tunnel is always rejected as a Release endpoint.

iOS is the only implemented app platform. Android and web do not appear as fictional choices. Linux can run the factory and API, but only macOS with Xcode can build and launch the iOS app.

Ownership and ejection

Remove the global CLI and an existing shot still has bun run verify, bun .tohseno/machine.ts, source, migrations, configuration, tests, and runbooks. No symlink points back to a factory cache. Identifiers, data paths, production references, and the static landing page remain with the shot.

Advanced and automation use

Humans can use tohseno or tohseno studio. Explicit commands including create, list, open, doctor, verify, adopt, run, and preview remain available for scripts and repository contributors. The detailed machine protocol is documented in each shot at .tohseno/OPERATIONS.md and in the repository.
